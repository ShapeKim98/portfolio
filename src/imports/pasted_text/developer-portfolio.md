## 소개

---

**RxSwift 기반 단방향 아키텍처 라이브러리입니다.**

ReactorKit은 Effect 개념이 없기 때문에 상태 변화에 따른 후속 작업을 정의하기 어렵다는 문제점이 있습니다.

이러한 문제점을 해결하기 위해, TCA의 단방향 아키텍처 매커니즘을 RxSwift 환경에 맞춰 구현했습니다.

실제 [**ShowPot**](https://github.com/AlreadyTakenSeat/ShowPot-iOS)이라는 프로젝트에 활용하여 적용 가능성 테스트 및 개선사항을 점검했습니다.

**구성원 :** iOS 1명

**담당 :** iOS 개발

## 기능 및 기술 설명

---

- **Action 및 State를 정의 하고, reducer를 재귀적으로 호출하기 위한 `Composer` 프로토콜 제공**
    - Action을 PublishRelay로 정의하여 Effect가 발생할 때마다 내부 map 연산을 통해 새로운 State를 생성하고, reducer 메서드로 재가공한 뒤 이를 바인딩하는 구조로
    구현했습니다.
    - Protocol 기반 제약을 도입하여, 개발자가 설계 의도에 부합하는 구조를 자연스럽게 구현하도록 유도했습니다.
- **새로운 Action 반환 및 Side Effect를 정의하기 위한 `Effect` 제공**
    - Effect를 associatedtype Action을 갖는 Protocol로 추상화하여, 다양한 Action 유형에 대응하는 Effect를 유연하게 처리할 수 있도록 설계했습니다.
- **상태기반 화면 전환을 위한 `@PresentState` 프로퍼티 래퍼 제공**
    - 모든 Output이 단일 State로 통합되어 있기에, 특정 프로퍼티 변경이 다른 프로퍼티에 영향을 미칠 수 있는 상황을 고려해 대응했습니다.
    - ReactorKit의 Pulse 개념을 참고해, 프로퍼티 래퍼 내부에 변경 카운터를 도입하여 해당 문제를 해결했습니다.
- **Action의 흐름을 자동으로 바인딩 하기 위한 `@Compose` 프로퍼티 래퍼 제공**
    - 초기 RxCompose는 Composer 프로토콜을 채택한 객체 생성 시마다 Action의 PublishRelay를 바인딩해야 하는 번거로움이 있었고, 이에 대응하고자 구성했습니다.
    - 프로퍼티 래퍼를 활용해 init(wrappedValue:) 시점과 wrappedValue의 set 시점에 자동으로 바인딩을 수행하도록 설계하여, 문제를 해결했습니다.
- **State를 Observable로 활용할 수 있게하는 `@ComposableState` 프로퍼티 래퍼 제공**
    - wrappedValue와 projectedValue를 활용해 BehaviorRelay를 값과 Observable 형태로 제공하도록 구성하여, 두 가지 사용 형태에 유연하게 대응했습니다.
- **Observable 및 Swift Concurrency 에 대응하기 위한 두가지 `run` Effect 지원**
    - RxSwift 기반이기 때문에 Swift Concurrency 기반 비동기 로직을 Reducer의 Effect로 변환하고자 구성했습니다.
    - 커스텀 Observable 내부에 Task를 구성하고, Observer를 파라미터로 받는 비동기 클로저를 실행하도록 설계하여, Observable 기반 Effect에 대응했습니다.
- **주기적인 액션 실행을 위한 `timer`, `interval` Effect 지원**
    - 별도의 Disposable를 받아 원하는 시점에 액션 실행을 종료할 수 있도록 구성했습니다.

CIFilter는 렌더링 흐름이 자동 최적화 방식으로 구성되어 있어 개발 편의성이 높지만 시스템 내부 처리 흐름을 개발자가 직접 제어하기 어렵기 때문에 실시간 반응성과 성능
미세조정 측면에서는 다소 제한적일 수 있다고 판단했습니다.
이에 따라 Metal을 통해 렌더링 파이프라인을 직접 구성하고, GPU 자원을 보다 직접적으로 활용함으로써 실시간 렌더링 성능을 확보할 수 있도록 구성했습니다.

### **필터값 실행 취소 및 다시 실행 기능 설계**

기능 구현을 위해 다음과 같은 요소들을 중심으로 설계를 진행했습니다.

- **필터값 이력의 순서 관리를 위한 Stack 구조 도입**
    
    필터값 상태는 스냅샷 형태로 저장되며, 각 스냅샷은 사용자가 슬라이더 조작을 마칠 때마다 캡처됩니다.
    작업을 최신순으로 쌓고, 필요 시 가장 최근 작업을 다시 복원하는 흐름은 Stack의 LIFO 특성과 유사하다는 점에서 Stack 구조를 도입했습니다.
    
    - 실행 취소 Stack: 사용자가 완료한 필터값 스냅샷들을 저장
    - 다시 실행 Stack: 사용자가 실행 취소한 필터값 스냅샷들을 보존
- **필터값 스냅샷 시점 전략**
    
    슬라이더 값을 조절할 때마다 필터값을 저장하는 방식은 반복적인 상태 기록이 누적되면서 성능 및 사용자 경험 측면에서 부담이 될 수 있다고 판단했습니다.
    또한, 조절 이력이 과도하게 쌓일 경우 ****원하는 시점으로 되돌리는 과정이 다소 복잡해질 수 있다는 점도 함께 고려했습니다.
    이에 따라, 필터값 스냅샷 시점은 사용자의 조작이 완료되는 시점인 DragGesture의 onEnded 메서드에서 수행되도록 구성했습니다.
    

## 채팅 기능 설계

---

### **로컬 DB 도입 배경**

채팅 내역은 한 번 저장되면 이후 변경되지 않는 정적인 데이터에 가까워, 채팅방 진입 시마다 동일한 데이터를 서버로부터 반복 요청하는 방식이 과연 효율적인지에 대한 고민이 
들었습니다.
더 나아가 로컬 DB를 도입 시 서버 트래픽 개선 뿐만 아니라 실시간 채팅 메시지 검색 시에도 서버를 거치지 않고 클라이언트에서 빠르게 조회할 수 있다는 가능성이 보였습니다.
이러한 점들을 종합적으로 고려하여 로컬 DB를 적용하는 방향으로 설계를 결정하게 되었습니다.

### **CoreData 선택**

프로젝트가 iOS 단독 플랫폼이며 iOS 16 이상을 타겟으로 하고 있다는 점을 고려했습니다.
CoreData는 비교적 낮은 iOS 버전에서도 사용할 수 있고, Apple에서 직접 제공하는 퍼스트파티 프레임워크라는 점에서 안정성과 호환성이 높다고 판단했습니다.
특히 채팅 기능에는 날짜 기반 커서 페이지네이션이나 검색어 기반 메시지 조회 등 비교적 복잡한 쿼리가 요구되므로 이를 지원하는 CoreData의 장점이 효과적이라고 생각했습니다.

### **DB 모델링 및 쿼리 구조 설계**

![coredata-schema-diagram 1.jpg](attachment:9b54972c-e7ac-44c5-a344-2040da0f01ac:3fc2ea19-dfd7-4706-8ee3-d78e9cf12b88.png)

- 채팅방에 속한 메시지와 참여자 목록을 효율적으로 조회할 수 있도록 채팅방은 채팅내역 및 참여자와 각각 1:N 관계로 구성했습니다.
- 채팅 내역은 분 단위로 연속된 메시지를 묶어 보여주는 UI 구조를 위해 각 메시지에 isHead, isTail 속성을 부여하여 저장 시점에 판단 및 기록하도록 구성했습니다.

### **클라이언트 모델과 동기화 전략**

- **서버와 로컬 DB 간 스키마 통합을 위한 DTO 변환 구조 설계**
    
    로컬 DB 구조와 서버 응답 스키마 간의 차이를 해소하기 위해 클라이언트 단에서 공통 모델을 정의하고 DTO를 통해 상호 변환하는 구조를 설계했습니다.
    
- **데이터 조회 효율을 높이기 위한 페이지네이션 고려**
    
    채팅 내역을 로컬 DB에서 조회할 때 전체 데이터를 한 번에 불러오는 방식의 효율성에 대해 고민하게 되었습니다. 이에 따라 필요한 범위만 조회할 수 있도록 페이지네이션
    방식을 직접 구현해 도입했습니다.
    
    채팅은 실시간으로 쌓이는 데이터 흐름을 갖고 있기 때문에 유연한 커서 기반 페이지네이션이 적합하다고 판단했고, 시간 순으로 정렬되는 데이터 특성을 반영하여 커서를
    날짜 기반으로 구성해 의도한 범위만 조회할 수 있도록 했습니다.
    

### **실시간 수신과 동기화 충돌 고려**

채팅방 진입과 동시에 서버와의 동기화 작업이 진행되는 동안에도 실시간 소켓 메시지가 수신될 수 있다는 점을 고려해야했습니다.

소켓 연결은 동기화보다 먼저 수행하고, 동기화 완료 이전에 수신된 메시지는 별도의 대기열(Queue)에 저장하도록 구성했습니다.
동기화가 완료되면 대기열(Queue)에 저장된 메시지를 선처리한 뒤 일반 수신 흐름으로 전환되도록 구현해 실시간성과 데이터 정합성을 보장했습니다.

### **채팅 검색 기능 설계**

검색 결과가 아직 불러오지 않은 메시지 범위에 포함된 경우, 곧바로 해당 위치로 스크롤 이동하는 데 어려움이 생길 수 있다는 점을 고려했습니다.
이에 따라, 현재 페이지네이션 커서의 날짜와 검색 대상 메시지의 날짜 사이 범위만큼을 먼저 조회한 뒤, 해당 메시지가 포함된 영역으로 자연스럽게 이동되도록 설계했습니다.

## 결제 기능 설계

---

### **결제 검증의 주체에 대한 고려**

결제 기능은 실제 청구가 발생하는 민감한 기능이기 때문에 기능 구현에 앞서 검증 방식에 대한 고민을 했습니다.

앱은 사용자 디바이스에서 실행되기 때문에 탈옥 등의 방법으로 앱을 임의로 변조해 결제 로직을 수정하고, 사용자가 인위적으로 ‘결제 완료’ 상태를  만들 수 있는 위험이 존재합니다.
반면 서버는 개발자가 통제하는 신뢰 가능한 환경이므로 모든 결제 검증은 서버에서 직접 수행하는 것을 원칙으로 했습니다.

### **결제 처리 흐름 설계**

1. 사용자가 필터 구매를 시도하면 앱은 서버에 주문 번호를 요청해 발급받은 후 PG(Payment Gateway) 시스템을 통해 결제 흐름을 시작합니다.
2. 결제가 완료되면 앱은 결제 승인 정보를 서버에 전달합니다.
3. 서버는 PG 결제 검증 API를 통해 해당 결제 건의 유효성을 확인하고, 금액, 상태, 주문 번호 등을 기준으로 검증이 완료된 경우에만 상품을 지급합니다.
4. 이후 클라이언트는 서버로부터 “결제 완료” 응답을 받은 후에만 UI 상태를 업데이트하고, 필터값 접근 권한을 부여합니다.

## 프로젝트 설계

---

![포킷그래프.jpg](attachment:aa7dd9aa-e410-4e6c-9f0c-c44f5e5d3cc1:포킷그래프.jpg)

### Modular Architecture 도입

- **기능 테스트 과정의 간소화**
    
    프로젝트 개발이 점점 고도화됨에 따라, 특정 기능을 검증하기 위해 여러 단계를 거쳐야 하는 상황을 개선하고자 했습니다.
    이에 테스트 흐름을 보다 간결하고 효율적으로 설계해 개발 과정의 집중도를 높였습니다.
    
- **빌드 효율성 향상**
    
    하나의 기능을 점검할 때 전체 프로젝트보다 해당 기능만 빌드하는 것이 더 효율적이라고 판단했습니다.
    이에 필요한 기능만 빠르게 빌드하고 실행할 수 있도록 모듈 단위 빌드 구조를 설계하여, 전체 빌드 시간을 단축하고 개발 효율을 높였습니다.
    
- **확장성과 재사용성 강화**
    
    프로젝트의 규모가 커질수록 유지보수와 확장을 유연하게 이어가기 위해,
    기능과 계층을 단순한 코드 단위가 아닌 독립적인 모듈 단위로 구성할 필요가 있다고 판단했습니다.
    모듈 단위 구성은 각 기능의 책임을 명확히 하고, 다른 코드와의 의존성을 줄여 관심사 분리(Separation of Concerns) 를 실현합니다.
    이를 통해 코드의 재사용성을 높이고, 다양한 기능을 일관된 구조 안에서 확장할 수 있는 기반을 마련했습니다.
    

### **Modular Architecture를 위한 Tuist 도입**

- **물리적 의존성 분리를 통한 책임 명확화**
    
    코드 레벨 분리는 폴더링이나 코드 관리만으로도 가능하지만, 실제로는 모든 코드에 접근이 가능해 개발자의 의도치 않은 참조나 책임 혼선이 생길 수 있다고 보았습니다.
    이에 타겟 기반으로 모듈을 분리해 접근 범위를 물리적으로 제한하고, 각 개발자가 접근 제어자를 통해 명확한 책임을 직접 관리할 수 있도록 설계했습니다.
    이를 통해 협업 시에도 일관된 구조와 책임 분리가 자연스럽게 유지되는 개발 환경을 구축했습니다.
    
- **Tuist를 활용한 모듈화 자동화**
    
    9개의 주요 기능을 독립된 모듈로 구성하고, 기능별 데모 앱까지 포함한 구조를 설계했습니다. 그러나 워크스페이스, 프로젝트, SPM 패키지, 디렉터리 구조를 일일이
    구성하는 것은 큰 리소스가 요구되었습니다. 이에 Tuist를 도입하여 Swift 코드 기반으로 프로젝트 생성을 자동화를 설계했습니다.
    모듈을 열거형(Enum) 으로 정의하고, 공통 설정을 Swift 메서드로 추상화하여 반복 작업을 최소화했으며, Tuist Template을 활용해 디렉터리와 필수 파일 구성을
    자동화했습니다.
    그 결과, 새로운 모듈 추가 시 열거형 케이스 한 줄 추가와 명령어 한줄이면 모듈 추가가 자동으로 완성되는 구조를 구축했습니다.
    
- **프로젝트 구조 시각화 및 유지보수 향상**

## URL 제목 및 썸네일 파싱 설계

---

링크 저장 서비스의 특성상, 사용자가 저장한 웹페이지의 썸네일 이미지와 제목을 안정적으로 파싱하는 것이 핵심 UX 중 하나였습니다.
이를 구현하는 과정에서 다음과 같은 세 가지 설계적 고민 및 문제 해결이 있었습니다.

### **사이트별 사용자 에이전트 요구 대응**

대부분의 웹사이트는 OG(Open Graph) 태그를 통해 제목과 썸네일 정보를 원활히 제공했지만, 일부 사이트에서는 요청 시 사용자 에이전트를 명시해야 정상적으로 데이터를
반환하는 사례가 있었습니다.
이에 사용자 에이전트를 명시적으로 지정하는 방식으로 문제를 해결했습니다.

### **OG 태그 미지원 사이트에 대한 UX 설계 제안**

OG 태그를 지원하지 않는 사이트의 경우 제목과 썸네일 정보를 파싱할 수 없었습니다.
이를 단순히 오류로 처리하기보다, 사용자가 직접 제목을 입력하고 기본 썸네일 이미지를 자동 삽입하는 UX 개선안을 PM에게 제안하여 사용자 경험 저하를 최소화했습니다.

### **썸네일 만료 이슈 분석 및 대응**

- **원인 파악을 위한 모니터링 및 가설 수립**
    
    썸네일이 로드되지 않는 문제가 보고된 이후, 팀원들과 함께 다양한 케이스를 재현하며 원인을 추적했고, 해당 현상이 인스타그램 URL에서 공통적으로 발생하는 패턴을
    확인했습니다.
    이후 동일한 인스타그램 URL을 매일 확인하며 응답 결과를 모니터링한 결과, 3~4일이 경과한 시점부터 썸네일이 표시되지 않는 현상이 나타났습니다.
    명확한 만료 주기를 확인하기 위해 인스타그램의 썸네일 만료 정책 관련 문서를 탐색했으나, 공식적인 자료를 발견하지 못했습니다.
    이에 내부 모니터링 결과를 근거로 썸네일 URL이 약 3~4일 이후 만료되는 것으로 추정했습니다.
    
- **JWT 토큰 리프레시 구조에 착안한 재파싱 로직 설계**
    
    썸네일이 일정 기간 후 만료된다는 가설을 바탕으로, **JWT 토큰의 발급–만료–갱신 흐름**에 착안한 재파싱 로직을 설계했습니다.
    썸네일 조회 실패 시 URL에서 새 이미지를 자동으로 파싱해 갱신하도록 구성하여, 사용자는 별도의 조작 없이 유효한 썸네일을 확인할 수 있도록 개선했습니다.
    
- **썸네일 갱신을 위한 서버 협업 및 API 개선 제안**
    
    사용자가 저장한 링크 데이터의 썸네일 URL을 갱신하지 않는 이상, 해당 URL은 만료된 상태로 유지되어 동일한 만료 URL을 계속 조회하게 됩니다.
    이로 인해 썸네일 조회 실패 → 재파싱 → 동일 URL 재조회가 반복되는 구조적 한계가 있었습니다.
    이를 방지하기 위해 새로 파싱된 썸네일 URL을 사용자 링크 데이터에 갱신하는 API를 서버 개발자에게 제안하여 협업을 진행했습니다.
    그 결과, 썸네일 정보를 안정적으로 유지하면서도 불필요한 재요청과 중복 파싱을 최소화한 구조를 완성했습니다.
